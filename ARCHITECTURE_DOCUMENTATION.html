🏛️ 尚书省 (Orchestrator)

定位: Python 状态机调度中枢，不依赖大模型

核心职责:
1. 驱动中书省和门下省完成"起草 - 审查"闭环
2. 维护全局黑板 (Blackboard) 存储任务执行结果
3. 按依赖关系顺序唤醒六部执行任务
4. 搬运上下文数据，实现上下文物理隔离

  • 容错循环: 最大 3 次重试，门下省驳回时强制中书省修正

    核心机制

    1. 起草 - 审查循环 (Draft-Review Loop)

    用户需求
        ↓
    【中书省】起草任务树 (Version 1)
        ↓
    【门下省】审查
        ├─→ ❌ 驳回 → 反馈 → 【中书省】修正 (Version 2)
        │                    ↓
        │              【门下省】审查
        │                    ├─→ ❌ 驳回 → ...
        │                    └─→ ✅ 通过 (最多 3 次)
        └─→ ✅ 通过 → 下发六部执行

    容错设计:

  • 最大 3 次重试

  • 每次驳回都带具体反馈

  • 重试用尽则终止任务

    • ---

    2. 黑板机制 (Blackboard Pattern)

    blackboard = {
        "T_01": {
            "department": "Hubu",
            "status": "success",
            "result": {"price": 15000, "params": {...}},
            "timestamp": 1234567890.0
        },
        "T_02": {
            "department": "Gongbu",
            "status": "success",
            "result": {"final_price": 17250},
            "timestamp": 1234567895.0
        }
    }

    优势:

  • 上下文物理隔离

  • 每个任务只拿到需要的数据

  • 省 Token，防幻觉

    • ---

    3. 确定性路由 (Deterministic Routing)

    尚书省用纯代码控制执行顺序

    sorted_tasks = topological_sort(task_tree)
    for task in sorted_tasks:
        # 1. 从黑板收集依赖数据
        context = gather_context(task["depends_on"])
        
        # 2. 唤醒对应部门
        result = call_agent(task["department"], context)
        
        # 3. 结果写入黑板
        blackboard[task["task_id"]] = result

    优势:

  • 不靠大模型猜测下一步

  • 拓扑排序保证依赖正确

  • 循环依赖会被检测并报错

    • ---

    4. 安全审查 (Security Check)

    工部准备执行代码
        ↓
    【兵部】静态安全扫描
        ├─→ AST 语法树分析
        ├─→ 正则模式匹配
        ├─→ 敏感信息检测
        └─→ 网络请求审查
        ↓
    判定风险等级
        ├─→ none/low → 放行
        ├─→ medium → 警告，需确认
        └─→ high → 阻断，上报

    ---

    5. 错误处理 (Error Handling)

    任何环节失败
        ↓
    【刑部】诊断错误
        ├─→ 可修复 → 提供方案 → 重试
        └─→ 不可修复 → 终止 + 人工介入

    ---

    六部无法解决的长尾需求
        ↓

        ├─→ 生成 System Prompt
        ├─→ 推荐工具
        └─→ 临时实例化专家
        ↓
    新专家继续执行

    完整工作流

    场景：数据中心报价

    1. 用户请求
       "客户需要 100kW 模块化数据中心报价，包含 2 台 50kW UPS 和 2 台精密空调。
        加 15% 利润后生成报价单并同步飞书。"
       
       ↓
       
    2. 🏛️ 尚书省唤醒 📝 中书省起草任务树
       中书省输出任务树 JSON
       
       ↓
       
    3. ⚖️ 门下省审查任务树
       审查结果：✅ 通过 (依赖正确、部门职能正确、闭环)
       
       ↓
       
    4. 🏛️ 尚书省拓扑排序 → [T_01, T_02, T_03, T_04, T_05, T_06]
       
       ↓
       
    5. 🛡️ 兵部安全审查 (代码扫描)
       审查结果：safe
       
       ↓
       
    6. 📦 户部执行 T_01, T_02
       查询 MySQL/Neo4j → 获取设备底价
       输出：{"UPS-50kW": 15000, "AC-20kW": 10000}
       写入黑板：blackboard["T_01"] = {...}
       
       ↓
       
    7. ⚙️ 工部执行 T_03, T_04
       从黑板获取 T_01, T_02 结果
       编写 Python 脚本计算：底价 + 利润 - 折扣 + 税费
       输出：{"final_price": 58477.50}
       写入黑板：blackboard["T_03"] = {...}
       
       ↓
       
    8. 🛡️ 兵部安全审查 (API 请求)
       审查结果：safe
       
       ↓
       
    9. 📤 礼部执行 T_05, T_06
       从黑板获取 T_04 结果
       生成 Markdown 报价单
       调用飞书 API 同步
       输出：{"document_url": "https://feishu.cn/docs/xxx"}
       写入黑板：blackboard["T_05"] = {...}
       
       ↓
       
    10. 🏛️ 尚书省返回最终结果
        {
          "status": "success",
          "blackboard": {...},
          "document_url": "https://feishu.cn/docs/xxx"
        }

    错误处理:

    任何环节失败
        ↓
    ⚖️ 刑部诊断错误
        ├─→ 可修复 → 提供修正方案 → 重试
        └─→ 不可修复 → 终止任务 + 人工介入

    长尾需求:

    六部无法解决
        ↓
    📋 吏部生成新专家
        ├─→ 分析领域知识
        ├─→ 生成 System Prompt
        └─→ 推荐工具
        ↓
    新专家继续执行

    总结

    三省六部制是一个完整的多智能体协作系统，包含:

  • 决策层 (三省): 尚书省 (调度)、中书省 (规划)、门下省 (审查)

  • 执行层 (六部): 户部 (检索)、工部 (计算)、礼部 (输出)、刑部 (诊断)、兵部 (安全)、吏部 (动态专家)

    • 核心优势:
    1. 起草 - 审查循环保证规划质量
    2. 黑板机制实现上下文隔离
    3. 确定性路由避免大模型幻觉
    4. 安全审查防止危险操作
    5. 错误诊断提供修复方案
    6. 动态专家处理长尾需求
