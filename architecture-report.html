2. 设计哲学

2.1 内外朝并行

优势： 轻任务走内廷（快速响应），重任务走外朝（严谨流程）。
维度 内廷 外朝
适用场景 闲聊、查进度、清理缓存 报价、写代码、发版同步
工作流 单点直连，快速响应 严格 DAG，多节点协作
模型选择 小模型 (Temperature=0.7) 大模型 (Temperature=0.0-0.3)
响应速度 极快 较慢 (多轮流转)
容错率 允许一定容错 0 差错，多次校验

2.2 尚书省纯代码化

核心洞察： 尚书省作为中枢神经，越"笨"（纯代码逻辑）越稳。

为什么不用 LLM 控制尚书省？

  • LLM 问题： 长对话后忘记状态、概率生成不够确定、每次调度消耗 Token、难以调试
  • 纯代码优势： Python 字典永远记住状态、确定性算法 100% 可预测、零 Token 消耗、完整日志追踪

2.3 代理人偏差防御

问题： 司礼监可能"矫诏"（过度总结、漏掉参数、脑补前提）

三大防御机制：

  1. 批红机制 (Human-in-the-Loop) - 司礼监拟旨后必须皇上确认
  2. 原文抄送 (Raw Context Injection) - 中书省同时看到原话和总结
  3. 三不铁律 (Prompt 降权) - 不准脑补、不准推理、宁可繁琐不可遗漏

5. 核心机制

5.1 上下文隔离 (_build_context)

效果： 最省钱、最防幻觉的设计。
def _build_context(self, depends_on): """只给当前任务需要的依赖数据""" context = {} for dep_id in depends_on: if dep_id in self.blackboard: context[dep_id] = self.blackboard[dep_id] return context # 示例：T3 只依赖 T2，绝不发送 T1 的数据 # 节省 70%+ Token，防止 Agent 看到无关信息胡思乱想

5.2 刑部自愈环 (_execute_with_xingbu_recovery)

效果： 系统不会挂掉，自动原地复活。
for attempt in range(3): # 最多重试 3 次 try: response = call_agent(department, payload) return response except Exception as e: if attempt == 2: raise RuntimeError("已达到最大重试次数") # 唤醒刑部诊断 xingbu_prompt = json.dumps({ "failed_department": department, "error_traceback": traceback.format_exc() }) xingbu_fix = call_agent("xingbu", xingbu_prompt) # 采用修复策略重试 payload["xingbu_fix_suggestion"] = xingbu_fix["fix_strategy"]

5.3 兵部静默钩子 (_call_bingbu_security_check)

效果： 恶意 Prompt 注入直接斩断。
# 在工部跑代码或礼部发包前，强制安检 if department in ["gongbu", "libu"]: bingbu_response = call_agent("bingbu", payload) security_report = json.loads(bingbu_response) if security_report["security_status"] == "dangerous": raise PermissionError(f"【兵部拦截】{security_report['block_reason']}")

5.4 黑板机制 (Blackboard)

class Blackboard: """共享状态存储 - 系统的内存池""" def __init__(self): self.data = {} # Python 字典 self.storage_path = "blackboard.json" self._load() # 从 JSON 加载 def set(self, key, value): self.data[key] = value self.save() # 立即持久化到 JSON def get(self, key): return self.data.get(key) # 特点： # 1. Python 字典永远记住状态 # 2. JSON 持久化，重启不丢 # 3. 所有 Agent 共享上下文
